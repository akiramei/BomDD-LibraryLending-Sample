# Charter — BomDD Library Lending Sample(forward-01)

> **実験の位置づけ**: BomDD playbook v1(フォワード・モード)の初回検証ループ。
> 方法論凍結版: BomDD repo `c0ab01f`(playbook v1 + gap-analysis v1 反映済み)。
> **Limitation(明記)**: 本ループはプロダクトオーナー役を設計者(AI)が代行する(要求エリシテーションの忠実性でなく、playbook の工程・ゲート・測定器の機能を検証するため)。Phase 5 の質問リスト裁定のみ実ユーザーが行う。

## 題材
- 何を作るか: 図書貸出 API(蔵書・会員・貸出/返却・延滞料金)。**プロセス再起動後もデータが残る永続化(SQLite)を含む** — N=3 が全て in-memory だった最大の未検証領域(gap-analysis §A3/§C2)。
- 種別: Web-API(.NET 10 minimal API + SQLite)
- 黒箱境界: wire(HTTP)。観測契約は不要。

## スコープ
- 含む: 蔵書登録 / 会員登録 / 貸出 / 返却 / 貸出一覧 / 在庫(部数)管理 / 貸出上限 / 期限・延滞料金 / 延滞ブロック / 再起動永続性 / NFR 1件(貸出レイテンシ)
- 含まない(理由付き):
  - 認証(webapi-0* で検証済みの表面。本ループは永続化に焦点)
  - 冪等キー(同上)
  - 予約・延滞通知(規模制約)
  - **スキーマ・マイグレーション**: グリーンフィールドの初回製造に旧データは存在しない=マイグレーションは**変更オーダー(Phase 7)の装置**である、を方法論判断として記録。forward-01.5(ECO: 会員区分の追加)で検証予定。
  - 同時要求の競合検査: 不変条件としては M-BOM に明記するが、固定オラクルでの検査は v1 では行わない(MeasurementCapability=insufficient-depth として記録。負荷・並行治具は次段)

## 工場構成(研究モード)
- ティア: 研究(初回 3 工場+探索プローブ)
- 使用モデル: opus / sonnet / haiku(クリーンなサブエージェント、model 上書き)
- 収束ループ予算: 2(seed → rev1 → rev2 まで)
