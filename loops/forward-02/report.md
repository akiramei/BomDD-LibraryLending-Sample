## 2. 主測定: 影響なし予測の成績(under-inclusion 1 / over-inclusion 0)
本 ECO は forward-01.5 の限界(3/3 unit 影響=絞り込みが試されない)への応答として、
**「依存があるのに影響しない」を切る予測**を反証可能な形で凍結した:

| 予測 | 結果 |
|---|---|
| S01–S25 全不変(standard 挙動・S25 は due 非検査) | **的中**(2工場 25/25) |
| 延滞ブロック・料金はコード不変(保存値経由の自動追従) | **的中**(S27/S28 が新期限への追従を確認・両工場とも当該コード無改修) |
| M01–M04 不変(スキーマ不変) | **的中** |
| Store.cs / Api.cs / Errors.cs / Ids.cs diff ゼロ | **的中** |
| **Program.cs 含む src/Library.Api 全体で diff ゼロ** | **反証** — 期限計算は Core の判定関数でなく Program.cs の別呼び出しサイトにあり、`ctx.MemberType` を通す配線1行が不可避。2工場が事前申告つきで**同一の1行に独立収束(git blob 同一)** |

粒度の分解: **E-BOM 粒度の予測(影響= E-DUE-FINE-001 のみ、1/12 品目)は的中**。外れたのは
品目→unit の写像 — 実装が1品目(期限計算)を Core と Api に横断分布させていた。
**教訓(method 還元候補)**: ECO の影響分析は BOM 上のトレース逆引きだけでなく、
**As-Maintained 実物の呼び出しサイト点検**(変更対象シンボルの全参照確認)を工程に含めるべき。
「実物が正」の影響分析版。予測を「diff ゼロ」のような反証可能な形で書いたからこそ、
取りこぼしが1行単位で可視化された(外れ方まで測定できる予測の書き方)。

## 3. 効力(effectivity)の初検証 — 未解決論点#2 への最初のデータ
- スキーマ変更なしの規則変更でも「既存データ上の振る舞い」は fixture でしか検査できない —
  **移行オラクルの一般化として effectivity-oracle(E01–E04)を新設**。
  fixture = v0.3 実ビルドで作成した DB(premium 会員の active 貸出=旧規則 +14日の確定値)。
- E02(既存貸出の期限は +14日のまま)・E04(料金は確定済み dueDateUtc 基準=遡及計算しない。
  FMEA-006 狙い撃ち)・E03(新規貸出のみ +21日)がすべて成立。
- 両工場とも遡及の失敗モードを作り込まなかった — `Fine`/`IsMemberOverdue` が確定済み DateOnly を
  引数に取る v0.1 以来の構造が、効力を**構造的に**保証していた(良い設計が保守の正しさを運ぶ実例)。

## 5. 結論
1. **「保守できる」の第2データ点**: 影響グラフの中心に入る変更でも、ECO+影響分析+効力オラクルは
   BOM に宿り fresh 2工場へ転移した(回帰・効力・不要改変すべてゼロ)。
2. **影響分析の限界が定量化された**: BOM 粒度の予測は的中し、実装分布(品目→unit 写像)の次元だけが
   外れた。是正は工程側(実物の呼び出しサイト点検)で可能 — 影響分析の精度は「BOM の正しさ」と
   「実物点検の有無」に分解できる。
3. **effectivity が方法論の語彙に入った**: 「規則変更が既存データに遡及しない」は移行と同格の
   検査対象であり、fixture ベースの専用オラクルで黒箱検査できる。
4. 限界: 依然 N=1 題材・単一 REQ 改訂・小規模コード。「非自明」は依存グラフの推論(3 hop の
   影響なし判定)についてであり、大規模題材の規模の壁(gap-analysis §C)は未検証のまま。
