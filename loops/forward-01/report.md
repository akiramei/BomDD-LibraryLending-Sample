## 3. ずる(全件は bomdd/51-cheat-log.md)
- 製造側: opus 9 / sonnet 8 / haiku 3(blocker 0)
- **検査器側 1 件(CHEAT-F01-H001)**: 治具が ConvertFrom-Json の暗黙 DateTime 変換に結合し opus を誤採点(19/20)→ `-DateKind String` で補正し 20/20。**v1.2「C2 は検査器に出る」がフォワードの初回採点でも再発(3例目)**。原版なしでは治具を事前にテストする相手がいないため、フォワードの治具リスクは構造的に高い(→手法改善候補: 治具のセルフテスト工程)。
- **手法側 2 件**: G2 再監査の省略(影響未観測)/ **自己受入の必須範囲が unit 行のみだったため haiku の API 表面欠陥が自己検査で見えなかった**(→手法改善候補: 必須自己受入に L1 API スモークを含める)。

## 5. 結論と次段
1. **フォワード・モードの中核主張が初回検証を通過**: 仕様→BOM→隔離製造で、原版なしでも「能力ある工場が初回 0 差分・残渣 0」に到達できた。G2/G3/K-BOM 事前抽出という「原版の代替装置」が、リバースで収束ループに払っていたコスト(webapi 2→3→0)を前倒しで払う形になった、という解釈が立つ(N=1。一般化しない)。
2. **二軸(決定性/正しさ)はフォワードでも機能**: haiku の脱落は BOM 補正では直らない(工場能力)。BOM 起因との分離が今回も可能だった。
3. 手法改善候補(BomDD 本体へ): ①必須自己受入に L1 API スモーク追加 ②治具セルフテスト工程(フォワードは原版がないため治具が最初の被検体になる)③G2 再監査の省略可否のテーラリング規準。
4. 次段: ユーザー裁定(質問リスト=report 末尾)→ 必要なら rev2 → **forward-01.5(変更オーダー+マイグレーション)**。
