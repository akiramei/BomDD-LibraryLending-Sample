## 1. 実験設計(なぜこの形か)

forward-03 で「部品交換+全再認証」が成立するケース(drop-in 互換)は実証済み。未観測だったのは
**交換が互換性で壊れるとき、処置がどう昇格するか**。実イベントは待てない(watch は緑)ため、
イベント種別として正当な**組織裁定**(SQLitePCLRaw 系列からの供給網離脱 — DEG-001 の実 GHSA が文脈)を
シナリオ起票し、非互換性は実在の部品差(Microsoft.Data.Sqlite ↔ System.Data.SQLite の API 差)に語らせた。

## 3. 発見

1. **昇格の決定点は scratch 事前検証に前倒しされる**。「適用後に全再認証が赤くなる」事象は
   発生しなかった — forward-03 で工程化した scratch 検証が決定的非互換(コンパイル/決定的挙動差)を
   先取りするため。適用後の回帰赤が残る領域は **scratch で再現できない非互換**
   (環境依存・非決定的・実データ規模依存)に限られる。「回帰赤→昇格」の実運用形は
   「**scratch 赤→昇格**」である(method 還元)。
2. **互換性の壁は K/M 層に閉じた**。REQ・仕様・E-BOM・オラクルは 1 行も変えずに部品交換+再製造が
   完結 = 挙動契約が調達部品に非依存であることの実証(BOM 層分離の検証データ)。
3. **知識パックが適応の裁量を吸収する**。K-SQLITE-001 v3 に型対応・パラメータ形式・PRAGMA 可用性を
   明記した結果、工場 cheat-report は **0 件(forward 系列で初)**。部品交換の適応は
   K-BOM の書き方次第で裁量ゼロの機械的作業に落ちる。
4. **データ互換性次元の初検査**: M01–M04 は「旧部品が書いた実 DB ファイルを新部品で開いて移行する」
   検査になった(32-mbom で out-of-scope 宣言だった「永続化(データ互換性)」の初の実測)— 合格。
   日時を TEXT 自前整形にしていた実装慣行(プロバイダの型変換に非依存)が互換を構造的に支えた。
5. 治具副観測(H 系列): V01 の健全性照合(raw 出力の言語マーカー要求)が日本語コンソールの
   符号化事故を**判定不能 fail として正しく検出**(偽緑防止ガードの実証)。恒久対処=
   治具内で `DOTNET_CLI_UI_LANGUAGE=en` を固定(ロケール非依存化)。
