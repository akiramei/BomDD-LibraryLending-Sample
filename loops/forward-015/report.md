# forward-01.5 報告 — 変更オーダー(ECO)の初回検証

> **一文要旨**: 納品済み個体(v0.2 = factory-04 ビルド)への変更オーダー(会員区分の導入+v0.2 スキーマ自動移行)を、影響分析→オラクル先行凍結→fresh 2工場の部分改修で実施し、**回帰 0・change miss 0・data-preservation miss 0・不要改変 0** を達成。**ECO と影響分析は BOM に宿り、別工場へ転移した**(観測した範囲)。

- 入力凍結: tag `forward-015-input` / 結果: tag `v0.3-forward-015`
- ECO 本体: [bomdd/60-change-order-eco-001.md](../../bomdd/60-change-order-eco-001.md)(§6 に測定記録)
- 裁定(2026-06-11): 工場=opus+sonnet の2(haiku は入れない)/ 非開示=オラクル実装・fixture 期待値 / 不要改変4分類 / 結果5分類 / 自己受入赤=stop/report

## 1. 結果(成功条件達成)
| 測定 | opus | sonnet |
|---|---|---|
| 回帰 S01–S23(不変ヤードスティック) | 23/23 | 23/23 |
| 変更受入 S24–S25 | 2/2 | 2/2 |
| 移行 M01–M04(fixture=実 v0.2 DB) | 4/4 | 4/4 |
| 不要改変(diff 基準点=複製コミット) | **0** | **0** |

- **影響なし予測が全的中**: S01–S23 の23ケース・非対象ファイル群とも(under/over-inclusion 0)。S07(既定 standard)も予測どおり不変——「既定値設計が既存オラクルを保存する」が成立。
- **移行方式が独立収束**: 2工場とも `PRAGMA user_version` + `ALTER TABLE ... DEFAULT 'standard'`(DROP/CREATE なし)。K-SQLITE-001 v2 に追記した移行規律が方式を一意化した(Loop4 の K-BOM 効果の保守版)。

## 2. Phase 7 の装置別検証
| 装置 | 結果 |
|---|---|
| 影響分析の先行凍結(影響なし予測=反証可能) | **機能した**(全的中。ただし N=1・小規模) |

| 部分改修の境界統制(diff 測定の事前宣言) | **機能した**(不要改変 0。「測定されると知っている」ことの規律効果を含む) |

## 3. ずる
- 製造側: opus 6 / sonnet 4(全 minor・blocker 0。user_version 選択・列の型・非文字列 memberType の扱い等、裁量宣言済み次元)
- 検査器側: CHEAT-F015-H001(M03 シナリオ日付の判定順序干渉。較正が凍結前に捕捉・修正)
- nonconformance: なし(両工場とも自己受入緑)

## 4. 結論
1. **BomDD は「作れる」に加えて「保守できる」の初回データを得た**: 変更オーダーが BOM(ECO+影響分析+K-BOM 移行規律)として記述でき、fresh 2工場へコーチングなしで転移し、既存挙動と既存データを保ったまま変更だけが入った。
2. 「PLM の価値は絞り込み」が要求変更側でも成立(Loop6 は外部知識劣化、今回は要求変更。突合相手=固定オラクル+実 DB という falsifiable な形で)。
3. 限界: N=1 題材・変更1件・小規模(4ファイル)。影響分析が自明に近い規模であり、影響グラフが深い大規模題材での再現は未検証(gap-analysis §C の規模の壁と同根)。
